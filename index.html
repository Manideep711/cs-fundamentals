Operating Systems

01 · Basics & Architecture 0:00 – 1:00:40

What is an Operating System?

📖 Definition

An Operating System is a system software that acts as an intermediary between the hardware and user applications/software. It manages hardware resources and provides a platform for programs to run.

💬 His Analogy

Think about how computers evolved — very basic machines first, where only things like editing were possible, and slowly more advanced versions came: Mac OS got more versions, Windows went 7 → 8 → 10 → 11, and Linux is the "base" everyone eventually has to learn if they're going into DevOps. The core aim of building a computer was always: I have hardware (RAM, ROM, storage) and I need the user to interact with it properly. Before all these OS options existed, people used the terminal directly on Linux to do everything. The OS's main job is making the hardware talk to the software — deciding where RAM goes, who gets it, how storage is managed, making sure no faults occur, and if there are multiple users on one machine, giving each one proper access only to their own files.

When your computer "hangs" and you have to reformat it, that literally means the OS stopped doing its job properly — formatting reinstalls a fresh, working OS, freeing up RAM in the process and making the system faster again. Every application you open (WhatsApp, Chrome, a YouTube app) is something a user runs — but what works internally to make this happen is the OS: system services, the file system, memory, I/O devices like the mouse/keyboard working correctly, files being allocated to the right folder structure, processes being scheduled, and the whole thing keeping in sync with the hardware.

📖 One-line core definition

OS = resource allocator and controller of programs. It controls programs and gives out resources based on what's needed.

🤔 Why does the OS even need to exist?

If the user came directly to RAM, they wouldn't actually care about RAM — they just want to run a simple app. That allocation work is the OS's job. Whether you run something via Linux or any other OS, the OS itself manages CPU, RAM, disk, and I/O devices directly, and hides hardware complexity from the user. The user doesn't manually decide "give this app 20MB of RAM" — the OS automatically pulls from whatever space is available and hands it over.

The OS provides: security, multitasking, memory management, and file handling. If the user had to do all of this manually, something would always get missed — so the OS handles security, lets you run multiple apps at once (multitasking) while keeping everything secure and properly spaced, and handles files.

📖 Primary goal of an OS

Make the whole system convenient and efficient. (If users had to do everything themselves, they wouldn't even open the app — people are "lazy" about even installing apps, let alone managing hardware requirements manually.)

Functions of the Operating System

⚙ Process Management

Creates, schedules, and terminates processes. Example: you run WhatsApp → OS creates the process, schedules it if needed, and terminates it when you properly close the app.

⚙ Memory Management

Decides how much RAM to give and when to reclaim it, based on available space and what the running app needs.

⚙ File Management

System files, directories, and permissions. If you have multiple user profiles on one computer, the OS builds separate directories so one user can't access another's files. Whoever logs in uses a password and gets only their own resources.

⚙ I/O Management

Handles devices like mice and keyboards directly.

⚙ Security & Protection

Authorization and access control (what we call authentication) — logging in with admin/root password on Windows or Linux.

⚙ Network Management

Internet connectivity, sockets, protocols, and port allocation — when you build web apps, the ports that get allocated are managed by the OS.

OS Architecture — The Five Layers

The OS architecture has roughly five layers. Bottom-most layer = hardware. Top-most = the user (where all your apps/programs sit). In between sit three layers whose job is to manage the OS itself.

Fig 1 — The 5-layer OS architecture. Kernel sits closest to hardware and is "always on."

System utilities directly interact with programs and the OS APIs, delivering whatever the user requires. The kernel sits just above hardware.

📖 What is the Kernel?

The kernel is closest to the hardware — it's the core of the OS. Think of it as the heart of the operating system.

🤔 Why must the kernel always stay resident in memory?

It is always resident in memory — always "on." If it isn't on, the OS fails and you need to reboot, or you go and run an OS/Windows update. The kernel watches every process, divides resources, and whenever you start a new program, an OS API call happens saying "this program has started," which goes to the kernel — and the kernel, being always active, manages the hardware. This also means it permanently occupies some space on the hardware itself.

The kernel handles: CPU handling, memory management, I/O management, and Inter-Process Communication (IPC) — IPC gets covered in more depth later in the video.

System Calls

📖 Definition

System calls are the interface between a user program and the OS. When an app needs a privileged operation, it makes a system call.

💬 His Analogy

It's just like calling an API and getting a response back — same idea, except here you're calling the OS instead of a web server. Whatever work the user wants done — execute, read, write, open, close, exit — gets requested via a system call. The app makes the system call, which goes into kernel mode to reach the hardware, the work happens, and the result comes back.

Types of Operating Systems (7 types)

📝 Note

"We've written it in seven parts. There are more types too, but this much is enough for your purposes."

1. Batch OS

📖 Definition

Jobs are collected, grouped into batches, and executed without user interaction. Example: IBM OS/360.

Jobs go into a queue → a batch monitor → CPU → output. You predefine tasks (Task 1, Task 2, Task 3), put them in a queue in a specific order, and a monitor tracks completion and output as each finishes. No direct user–CPU interaction; everything executes without user intervention.

2. Multiprogramming vs Multitasking vs Multiprocessing

🔥 Direct interview question

"This is asked directly in interviews — explain the difference between these three."

📖 Multiprogramming

Multiple programs loaded into memory simultaneously, but only one process actually runs at a time — the CPU picks up Process 1, completes it, switches to Process 2, completes it, then moves to Process 3.

📖 Multitasking (Time-Shared OS)

An extension of multiprogramming, but the CPU switches rapidly between processes, giving the illusion of parallelism. Example: Windows.

💬 His Analogy

It feels like everything is running together, but actually the CPU is swiping between tasks — it's switching so fast that it looks like multitasking, but it's really still multiprogramming under the hood, just at extreme speed. Each process gets a time quantum (e.g. 10 milliseconds) — so in 1 second the CPU might switch ~100 times, giving you the feeling everything is happening at once when actually the CPU is doing one thing at a time, just blazing fast.

📖 Multiprocessing

Multiple CPUs in one system → true parallelism — multiple processes genuinely run together on different CPUs at the same time.

3. Real-Time OS (RTOS)

🔥 Often asked directly

RTOS has strict time/complexity constraints and deadlines must always be met (Hard RTOS). Soft RTOS is less critical about deadlines — a little lag is acceptable. Examples: FreeRTOS, VxWorks.

4 & 5. Distributed OS and Embedded OS

Distributed OS: multiple networked computers grouped together appear as one system (e.g. Google's system, Hadoop clusters) — you can run 5-6 different computers/OSes as one combined operating system.

Embedded OS: built for specific hardware — e.g. a microwave has some processing happening inside it, a smart TV runs a different app and needs its own OS because it's hardware too (it has some RAM, some space, Wi-Fi connectivity). Android is itself an extension of this — your phone's camera needs its own hardware-access process, so we use an embedded OS for that. Most modern devices use embedded OS.

Process Management

📖 What is a Process?

A process is "a program in execution." A program that is running is called a process. It is an active entity — active because it's currently executing — and it has its own memory, space, state, and resources.

Process Control Block (PCB)

📖 Definition

Whenever a process runs, it needs memory, space, files — a proper own resource set — and all of this active-state data is stored in a structure called the Process Control Block.

🔥 Direct exam/interview question

"This was directly asked in my own college exam — 'what are process states?'"

PCB stores: Process ID, Process State, Program Counter, CPU Registers, Memory Management info, I/O status, Accounting info.

Context Switching

📖 Definition

When the CPU switches from one process to another, this is called context switching. The current process's PCB is saved; the CPU then loads the next process's PCB and resumes from there.

Sequence: Process 1 running → save P1's PCB → load P2's PCB → run P2 → (need to switch back) → save P2's PCB → load P1's PCB → resume P1. The extra time spent on all this saving/loading/switching (not on actual computation) is called pure overhead.

CPU Scheduling

🔥 Very important for interviews

"Again I keep repeating it — everything we're discussing in this video is a very important topic, and you need to make proper notes."

🤔 Why does scheduling exist?

Multiple programs and threads are running. You need to decide which task runs next, switching between them in milliseconds. Your goal should be: maximize CPU utilization, minimize waiting time. That's CPU scheduling — deciding which process runs next on the CPU.

Scheduling Queues

  • Job Queue — all processes in the system.
  • Ready Queue — processes in RAM, ready to execute.
  • Device Queue — processes waiting on an I/O device (e.g. waiting for the printer or keyboard input).

Types of Schedulers

SchedulerJobFrequency
Long-Term (Job Scheduler)Decides which program gets loaded into RAM; manages degree of multiprogrammingInfrequent — seconds/minutes
Short-Term (CPU Scheduler)Decides which ready process goes into the CPU nextVery frequent
Medium-Term (Swapper)Swaps processes in/out of memory (I/O — in and out of memory)Periodic

Scheduling Criteria

  • CPU Utilization — keep it maximum (aim ~40–90%).
  • Throughput — number of processes completed per unit time; should come out well.
  • Turnaround Time (TAT) — submission to completion time; should be minimum.
  • Waiting Time — time spent in the ready queue; lower = faster switching.
  • Response Time — time from submission to the first response; should be fast.

Scheduling Algorithms

Two broad categories: Non-preemptive and Preemptive.

Non-Preemptive Algorithms

1. First Come First Serve (FCFS)

📖 Definition

The process that arrives first gets the CPU first. Simple queue behavior.

⚠ The Convoy Effect Problem

A short process can end up waiting behind a long one. If a 1-minute job and a 10-minute job both need the CPU and the 10-minute job arrived first in the queue, the 1-minute job has to wait the full 10 minutes — even though it's tiny — because FCFS doesn't care about job size, only arrival order. This is the convoy effect.

Gantt Chart

📖 What is a Gantt Chart?

A time-vs-process graph: x-axis = time, blocks = which process is running during that time. Process 1 runs 0→5 (if it arrived first and took 5ms), Process 2 runs next, and so on, based on arrival time.

2. Shortest Job First (SJF)

📖 Definition

Non-preemptive. The process with the shortest burst time gets the CPU first.

⚠ Starvation Problem

If short processes (e.g. 5-minute jobs) keep arriving repeatedly, a long process (e.g. a 10-minute job) might never get to run — every time the scheduler picks "shortest job," a newer short job keeps cutting in line. The big process starves: requires knowing burst time in advance, and causes starvation for long processes.

Example walked through: arrival/burst times for 3 processes → 2-second job runs first, then the 4-second one, then the 6-second one (shortest-burst-first), and average wait time is computed by summing wait times ÷ 3.

Preemptive Algorithms

3. Shortest Remaining Time First (SRTF)

📖 Definition

The preemptive version of SJF. If a new process arrives with a burst time shorter than the remaining time of the currently running process, the CPU preempts (interrupts) the current process.

⚠ Trade-off

Optimal in theory, but causes high context-switching overhead. Example: a 10-minute job runs for 2 minutes, then a new 2-minute job arrives — the scheduler breaks the running job, finishes the new 2-minute job first, then resumes the remaining 8 minutes of the original. Those two breaks each cost context-switch time, adding overhead.

4. Round Robin (RR)

📖 Definition

Every process gets a fixed time quantum (e.g. 2 seconds), runs for that long, then moves to the next process — cycling through all processes. Used in time-sharing systems. When a process finishes early, it terminates and the cycle continues among whatever's left.

🤔 The Time Quantum Trade-off

If the quantum is too small → too many context switches (overhead dominates). If the quantum is too large → it behaves like FCFS (if you give a 1-hour quantum, the first process basically runs to completion before switching). If the quantum is very small, it can start resembling Shortest-Job-First-like behavior because of so much switching — not exactly SJF, but context switching becomes very heavy.

Worked example: with a 2-second quantum across 3 processes, Process 1 runs 2s, Process 2 runs 2s, Process 3 runs its full burst, then back to whoever has remaining time — average waiting time computed out to ~3.3 in the example.

5. Priority Scheduling

📖 Definition

Each process gets a priority number. Lower-priority processes run last; higher-priority run first. By convention, priority numbers increase as priority decreases — priority 1 = highest, priority 100 = lowest (like a ranking system). Can be preemptive or non-preemptive.

⚠ Starvation + Aging Fix

Low-priority processes can starve forever if high-priority ones keep arriving. The fix is Aging: gradually increase a waiting process's priority over time (e.g. +1 every time another process completes) — so eventually, even a priority-100 process climbs down to priority-1 and finally gets to run.

6. Multilevel Queue Scheduling

The ready queue is divided into multiple separate queues (system, interactive, batch), and each queue has its own scheduling algorithm — you can apply different algorithms to different queues based on need (e.g. high priority gets its own queue and its own algorithm).

7. Multilevel Feedback Queue

Like multilevel queue, but processes can move between queues. This is the most flexible and most complex to implement, and is used in most common modern OS like Linux CFS (Completely Fair Scheduler).

Process Synchronization

The Critical Section Problem

📖 Definition

When multiple processes access shared data concurrently, we get a race condition. The critical section is the code segment that accesses the shared resource.

💬 His Explanation

If one piece of data exists and multiple threads/processes want to read/write it at the same time, a race condition occurs — who gets it first, in what order, becomes the problem. That's why we need solutions for the critical section.

Requirements for a Critical Section Solution

  1. Mutual Exclusion — only one process in the critical section at a time.
  2. Progress — if the critical section is empty, a waiting process should be allowed to enter.
  3. Bounded Waiting — a process must be able to enter the critical section within a finite time (no infinite starvation) — so no other process should make it wait forever.

Peterson's Solution

Pseudocode approach: each process gets a flag (true/false) and there's a shared turn variable. If process J's flag is on and it's J's turn, it can enter the critical section. After it's done, it sets its flag to false, letting the next process run, and so on — for two processes, satisfying: flag[i] is true, turn == i, and flag[j] is false.

Locks & Synchronization Primitives

Mutex (Mutual Exclusion Lock)

📖 Definition

A binary lock: 0 = unlocked, 1 = locked. Only the thread that locked it can unlock it.

⚠ Busy waiting / spin lock

While a resource is locked (value 1), other threads stay in a "busy wait/spin lock" loop until it becomes 0 (unlocked) again.

Semaphore

📖 Definition

An integer variable accessed via atomic wait() and signal() operations. More powerful than a mutex. wait() decreases the value; signal() increases it. If the value goes negative, the process is blocked; if value ≤ 0, the OS may wake up a waiting process; if signal makes it ≥ 0, the process is allowed to proceed.

TypeValues
Binary Semaphore0 or 1 (behaves like a mutex)
Counting SemaphoreAny positive integer
💬 The Printer Analogy

Say you have 5 printers → semaphore = 5. Each print job that comes in does wait(), so S becomes 4, 3, 2, 1, 0 as jobs use up printers one by one (Printer 1, Printer 2, Printer 3... up to Printer 5). After a print finishes, signal() increases S back up. If S hits 0, the process must wait until another printer frees up. After all printing is done, the value rises back to 5 via repeated signal() calls — printers become available for the next batch of jobs.

Mutex vs Semaphore Walkthrough

📝 How locking actually plays out

With a mutex: value 1 = locked, value 0 = unlocked. When Thread 1 finishes, you set it to 0 (unlock), letting Thread 2 or Thread 3 (whoever needs it) take the critical section, finish their work, then release it back to 0 when done. With a semaphore: same idea but using wait/signal increments and decrements, and it generalizes to multiple resources (like the 5-printer example) instead of just one lock.

Classical Synchronization Problems

1. Producer–Consumer Problem

📖 Bounded Buffer Concept

A producer adds items to a buffer; a consumer removes items. The buffer has a fixed size n.

💬 His Analogy

If you have storage for 4 items and you're the producer with 3 consumers currently, you fill the buffer with items 1, 2, 3 — then as space frees up, you refill it so a new consumer can take directly. If you suddenly have 5 consumers, the 5th has to wait until one of the first 4 slots is used up and freed. So with 4 RAM-like slots, you allocate to 4 processes, some wait, and as soon as one finishes, the 5th consumer can come in and use the freed slot.

2. Readers–Writers Problem

📖 Definition

Multiple readers, one (or more) writer. The danger: data might get read before it finishes being written, or while it's still being written — this produces stale data.

One standard solution version: no reader waits unless a writer is currently writing. The writer gets priority access; readers get priority only once the writer is done.

3. Dining Philosophers Problem

💬 His Setup

5 philosophers at a table, 5 chopsticks, and each philosopher needs 2 chopsticks to eat. This creates a potential deadlock situation. In practice, ~4 philosophers might sit and eat first; as soon as the next chopstick frees up, the next philosopher in line can eat.

03 · Deadlocks 1:12:00 – 1:19:11

What is a Deadlock?

🔥 Very important from an interview standpoint
📖 Definition

A deadlock is a situation where a set of processes are blocked because each process is holding a resource and waiting for another resource to be released (held by some other waiting process), so it can also use it.

💬 The Metro/Bus Seat Analogy

It's like traveling on a metro: when a seat is empty, you sit. Until then, you wait. You're in a "deadlock-like" condition until the seat frees up. And — important nuance — even if someone got on after you, you can't shove them out of the seat just because you were waiting longer; everyone forms a line and the rule is strict ordering: whoever's turn it is, sits when it's available. Nobody jumps the line, and nobody is forcibly removed from a seat. ("Everyone paid for their ticket.")

🔥 Coffman's Four Necessary Conditions for Deadlock

🔥 Memory trick: "MHNC"

Mutual Exclusion → Hold and Wait → No Preemption → Circular Wait.

  1. Mutual Exclusion — a resource is held by only one process at a time.
  2. Hold and Wait — a process holds a resource while waiting for another (like waiting for a seat while still standing in your spot in line).
  3. No Preemption — a resource can't be forcibly taken away (you can't shove someone off a seat).
  4. Circular Wait — P1 waits for P2, P2 waits for P3, P3 waits for P4 ... forming a circular dependency chain (like a properly-ordered queue where the person who got on after you waits behind you, and you wait behind whoever's ahead — a circular waiting structure).
📝 Remember the metro example

All four conditions must hold simultaneously for deadlock to occur.

Resource Allocation Graph (RAG)

📖 Definition

A process has a request edge towards a resource it wants, and the resource has an assignment edge towards the process once granted.

🤔 Detecting deadlock via RAG

If the RAG has a cycle with single-instance resources, it confirms a deadlock. With multiple-instance resources, a cycle is necessary but not sufficient for deadlock — meaning a cycle might exist without an actual deadlock if there are spare instances of that resource.

Deadlock Handling Strategies (4 strategies)

1️⃣ Deadlock Prevention

Remove one of the four Coffman conditions. E.g. "no mutual exclusion" → make resources shareable (not always possible). "no hold-and-wait" → request all resources at once before starting. Allow preemption → if a resource is needed, preempt it from a waiting process. No circular wait → assign a strict ordering to resource types.

2️⃣ Deadlock Avoidance

The OS checks in advance whether allocating a resource will leave the system in a safe state — a safe state is one where a "safe sequence" exists in which all processes can complete. This uses the Banker's Algorithm (checks availability, max need, and current allocation before granting).

3️⃣ Detection & Recovery

Allow deadlock to occur, then detect it (run a detection algorithm) and recover — either kill all deadlocked processes and restart, or selectively kill/reallocate.

4️⃣ Ignorance — The Ostrich Algorithm

Used when deadlocks are rare and the cost of prevention is high. Linux and Windows mostly rely on this in practice.

Memory Management

🤔 Why memory management matters

Multiple processes run in RAM simultaneously, and we need to allocate and deallocate memory so everything runs fast and properly, and so one process doesn't interfere with another (interference = data isn't safe, processes mess with each other's memory and things stop working correctly).

Memory Hierarchy

💬 His Analogy

Imagine a staircase you built like little boxes stacked downward — the lower you go, the cheaper and slower it gets. "Cheap = slow."

Fig 4 — Memory hierarchy pyramid (his "staircase" mental model)
  1. Registers — fastest, smallest, inside the CPU itself, measured in bytes.
  2. Cache memory — works in nanoseconds.
  3. RAM (main memory) — 4GB, 8GB etc.; more RAM = more expensive laptop.
  4. Secondary storage (SSD/HDD) — works in milliseconds; per-GB cheaper than RAM (e.g. going from 512GB to 1024GB storage costs far less than going from 4GB to 8GB RAM).
  5. Network — slowest; pulling data over 4G/5G is the slowest tier of all.

Logical vs Physical Address

📖 Definitions

Logical address = generated by the CPU; this is the "virtual" address. Physical address = the actual address in RAM (the chip itself). The Memory Management Unit (MMU) maps logical → physical addresses.

Contiguous Memory Allocation

📖 Fixed Partitioning

RAM divided into fixed-size partitions.

💬 The Cake Analogy

Imagine a big cake split into two equal parts. A small kid who's not very hungry still gets the same equal-sized piece — the extra goes to waste. But you always divide equally regardless of who needs how much, so small processes waste space, and big processes that need more run into trouble fitting.

📖 Dynamic Partitioning

The exact size needed gets used, but memory becomes scattered — total free space adds up, but it's fragmented across the system.

Paging

📖 Definition

Physical memory is divided into fixed-size frames. Logical memory is divided into fixed-size pages. The logical address maps to the physical address through the page table — same idea as the MMU mapping mentioned above.

The OS builds a page table that maps each process's page number → frame number. A logical address = (page number, offset); the page-to-frame mapping plus the offset gives you the final physical address.

Multilevel Paging & Inverted Page Table

  • Multilevel Paging — with large address spaces, the page table itself becomes too large, so it's split into multiple levels.
  • Inverted Page Table — one entry per physical frame instead of per logical page → shrinks the table size significantly.

Segmentation

📖 Definition

Memory is divided into variable-sized segments based on logical units — code, stack, heap, data — each segment is mapped to a base address + limit (size).

🤔 How addressing works

If you know a segment's address and its size, you can directly reach the next address — that's how you can jump to the next segment based on the current segment's known span.

Virtual Memory

📖 Definition

A technique that allows execution of processes that are not completely in physical memory. The disk (e.g. SSD space) is used as an extension of RAM — an illusion of having more memory than physically exists.

Demand paging loads pages into RAM only when actually needed — if the page being accessed isn't in RAM, it's fetched from disk, causing a page fault.

Page Fault Handling

  1. OS detects the invalid access.
  2. Finds a free frame in RAM.
  3. Loads the page from disk into that frame.
  4. Updates the page table.
  5. Restarts the instruction.

Page Replacement Algorithms

AlgorithmLogic
FIFOReplace the oldest page in memory
OptimalReplace the page that won't be used for the longest time in future
LRU (Least Recently Used)Replace the page that's been unused for the longest time
Clock AlgorithmApproximates LRU using bits and a circular ("clockwise") sweep over candidate pages

Thrashing

📖 Definition

The process spends more time paging than actually executing.

💬 The "Chicken vs Spices" Analogy

"Murgi se zyada masale ka kaam ho gaya" — you spent more on the spices/seasoning (₹200) than on the actual chicken (₹100). What's the point? Similarly, if managing all these processes ends up taking more time than the main work itself, that's thrashing — too many processes get too few frames, forcing constant disk-swapping, and CPU utilization (and effectively multiprogramming) collapses.

File Systems

📖 What is a File?

Data stored in secondary storage — PDFs, images, videos, text, binary, executable code, or even directories (folders).

File attributes: name, type, location, size, creation/modification time, who's accessing it — all visible when you open File Explorer (and you can rename, etc.). File operations: open, write, delete, close.

🤔 Why I/O needs careful management

Device drivers (mouse, keyboard) must be managed properly so one program using a device doesn't interrupt another, and direct memory access must be controlled correctly.

Storage & Caching

📖 Cache

Small, fast memory sitting between CPU and RAM — directly accessible by the CPU, hence extremely fast. Cache levels: L1, L2, L3 — L1 is inside the CPU core (smallest size); L2/L3 sit progressively further out (bigger, slightly slower).

📖 Cache Hit vs Cache Miss

Cache hit = found in cache. Cache miss = not found → must be fetched and then stored in cache so the next access becomes a hit. Estimated Access Time formula: h × cache_time + (1-h) × miss_penalty where h = hit ratio.

Protection & Security

The OS must always prevent unauthorized access — only the intended user can access their own data. Different users get different privileges and access is allocated accordingly. Authorization and Authentication are both critical, along with Auditing (keeping logs).

Access Control List (ACL) / Unix Permission Bits

📖 The 9-bit permission structure

Permissions are grouped: Owner, Group, Others — each gets 3 bits for Read, Write, Execute (RWX), totaling 9 entries. Example: rwxr-xr-- means owner has read/write/execute, group has read/execute, others have read only.

📖 Three Security Concepts
  • Authentication — "Who are you?" (e.g. a password check — like proving you're actually a teacher before you're allowed in).
  • Authorization — "What can you do?" (e.g. being allowed to mark attendance once you're verified as a teacher).
  • Auditing — keeping logs of everything, so the system can be debugged later if something needs investigating.

Threat types: viruses, worms, trojans, ransomware, rootkits — malicious code can introduce all of these problems into an OS.

Virtualization & Containers

📖 Virtual Machines

You can create virtual sessions of hardware and OS — one physical hardware running multiple smaller "machines." This is how, e.g., you run Linux on top of Windows, or Mac on top of Windows (VMware-style). A full OS runs in its own carved-out memory space and treats it as if it were the main hardware.

📖 Containers

Run on a shared kernel — much lighter than a VM (e.g. Docker). A container runs your code on the current OS directly (sharing the host kernel), unlike a VM, which spins up an entirely new OS instance to run on.

Linux is the most popular kernel choice generally — it uses a Red-Black Tree internally (in its CFS scheduler) and is very lightweight, which is why it's used so widely.

Summary: Boot Process, Kernel/User Mode, Interrupts

  • Boot Process — what happens at power-on, the whole startup sequence.
  • Kernel Mode vs User Mode — user mode has restricted access (programs run here); kernel mode has full hardware access and runs the OS itself.
  • Interrupts — a signal from the keyboard is a hardware interrupt; a system call is a software interrupt; a failed condition is an exception.

What is a Computer Network?

💬 His Analogy

Look around your home or office — lots of computers, phones, tablets, smart watches, TVs — all connected to each other via Wi-Fi. That's why apps exist that transfer data over Wi-Fi directly: you can transfer videos/movies directly over Wi-Fi without using mobile data or even an internet connection at all. If your printer is on the same Wi-Fi, files go straight from your PC to it.

📖 Definition

A computer network is a collection/group of interconnected devices sharing data and resources with each other. "Network" itself just means a collection or group — a Computer Network means a group of computing devices, connected, sharing data and resources.

🤔 What actually counts as a "network"?

Just having a wire plugged in but switched off does not count as a network — devices must be actively exchanging data/resources (e.g. a printer that's actually receiving and printing a file is participating in the network; a powered-off device is not).

Types of Networks (by size/range)

How Data Actually Travels — His End-to-End Walkthrough

💬 The YouTube/Google Server Journey

Imagine a main server (Google's/YouTube's server). It connects via wired connections that run as undersea cables to reach a country (e.g. India), arriving at a local server within that country. When you search www.youtube.com, DNS resolves it (explained later), and the request travels from the main server → local server (India) → through multiple towers → to your nearest tower → to your home's Wi-Fi router → from there it goes wireless all the way to your phone. The whole journey happens within milliseconds. The person who engineers this speed is a network engineer — their job is to make this happen in milliseconds using optical fiber, which transfers millions of GB of data per second across huge distances. Up to the Wi-Fi router it's wired; from the router onward it goes wireless. (If you plug a LAN cable directly into your laptop instead of using Wi-Fi, it gets even faster because the connection becomes truly wired all the way through.)

02 · OSI Model 1:59:41 – 2:16:58

🔥 Most-asked CN interview topic

"Most of your interview questions for CN will come from the OSI model — if you're going for interviews and CN questions come up, they'll mostly be from the OSI model."

📖 Definition

OSI = Open System Interconnection. It defines how data travels through 7 distinct layers.

His "Sending a Hi Message" Walkthrough

💬 The Full Journey, Layer by Layer

Say you typed "Hi" and sent it to someone. First, you typed it on your phone — that's Physical Layer work (a physical message converted into bits). The typed message goes to your Wi-Fi router — still Physical Layer because it travels via cable/hub. The moment it reaches the Wi-Fi/router and routing decisions happen about which device it's going to — that becomes Data Link Layer, then onward into Network Layer work (IP addressing, deciding which device to route to). Then comes the Transport Layer — a protocol decides whether this message is going to one device or broadcasting to multiple devices. Once it reaches the destination phone, the Session layer starts (e.g. your 4G connection session begins, like that "connecting" handshake). Then Presentation Layer handles SSL certification/encryption/authentication checks. Finally, the Application Layer is where you actually see the "Hi" message appear on the screen via HTTP/the app's UI.

03 · TCP vs UDP & Protocols 2:16:59 – 2:23:19

TCP vs UDP

💬 The WhatsApp vs SMS Analogy

You send someone a message on WhatsApp — first a single tick (sent), then a double tick (delivered), then it turns blue (read). Now you can confidently say "you read the message but didn't come" — that's TCP behavior, because you get acknowledgment at every step. But if you send a plain SMS, there's no "read receipt" feature — you never get confirmation the other person actually saw it. That's UDP: no acknowledgment that your message even arrived, let alone was read.

📖 Definitions

TCP (Transmission Control Protocol) = connection-oriented, reliable, acknowledgment-based — but slower (multiple round trips). UDP (User Datagram Protocol) = connectionless, no acknowledgment — but faster.

The Three-Way Handshake (TCP)

Fig 8 — TCP's 3-way handshake: SYN → SYN+ACK → ACK

His step-by-step description: client says "I want to connect" → server replies "SYN + ACK" (yes, let's connect, acknowledged) → client sends a final ACK confirming, and then actual data transfer begins. This 3-step process is exactly why TCP is slower — three round trips happen before any real data moves.

Protocols & Ports

💬 The Road Rules / Security Gate Analogy

The internet has rules just like road traffic rules: drive on the correct side, stop at red lights, wear a helmet — break them, get fined. Protocols are the internet's version of these rules — data must be encrypted, one device shouldn't be able to peek into another's data, etc. If you violate these protocols deliberately, that's how hacking happens — you stop following the rules. Ports work like a security gate at an entrance: just like you need a valid license/helmet to pass through a checkpoint, a port has security checks that validate headers before letting data through; if something's wrong, the data gets bounced right there.

📖 What is a Protocol?

A set of rules that devices on a network must follow so that communication stays orderly, secure, and interpretable by both ends.

04 · DNS & Routing 2:23:20 – 2:38:57

DNS — Domain Name System

💬 The "Why Not Just Memorize IPs" Analogy

When you open facebook.com, you remember "facebook.com" easily. But imagine if instead you had to memorize and type a raw IP address every single time, for every one of the ~50 sites you visit regularly (YouTube, Maps, Google, Yahoo, Bing). That would be a nightmare. So a lookup table was built: IP on one side, Domain Name on the other — and that table/system is the DNS — Domain Name Server.

📖 Definition

DNS converts human-readable domain names into machine-usable IP addresses, via a central database. When you buy a domain (from GoDaddy etc.), it gets registered with a managing body — that's also why domain registration can take a day or two.

DNS Resolution Flow

Fig 9 — DNS resolution: domain name → IP → connection → page render

His full sequence: you type google.com in the browser → request goes to DNS → DNS returns the IP address → a TCP connection is built → an HTTP request is made → the request goes to the server at that IP → the server sends back a response → the browser renders it on your current IP/device.

DHCP — Dynamic Host Configuration Protocol

📖 Definition

Automatically assigns an IP address, subnet mask, gateway, and DNS to a device when it joins a network — modern routers do this automatically so your device connects to the internet without manually configuring anything.

🔥 DORA Process

🔥 Direct question — memorize "DORA"
Fig 10 — DHCP's DORA handshake (his "new phone joins Wi-Fi" example)

His phrasing of the analogy: a new device connects to a network and asks (via Wi-Fi broadcast) "is there an IP available?" The router/Wi-Fi responds "yes, here's an IP, take it." The client requests it ("okay, I'll take this IP") and the server finally acknowledges ("confirmed, you've been allotted this IP — now you can use the internet"). This entire DORA flow runs automatically every time a phone joins a Wi-Fi network.

ARP & RARP — Address Resolution Protocol

📖 Definition

ARP converts an IP address → MAC address. RARP (older, mostly replaced by DHCP now) does the reverse: MAC → IP.

💬 His Router-Table Analogy

Say data arrives at your Wi-Fi router addressed to one shared IP, but 10 devices are connected to that same Wi-Fi. The router keeps an ARP table mapping physical (MAC) addresses to network addresses, so when a response comes back for "this IP," the router knows exactly which physical device (by MAC address) actually requested it — "this is the one that asked for this data" — and forwards it precisely there using ARP.

MAC Address

📖 Definition

The hardware address of the NIC (Network Interface Card) — 48-bit, unique to every device. Unlike an IP address (which can be shared across a network of 10 devices behind one router), the MAC address is uniquely tied to one specific device.

NAT — Network Address Translation

📖 Definition

Converts a private IP address to a public one (and back) so multiple devices on a private network can share one public-facing connection to access the internet.

💬 His Walkthrough

You have one Wi-Fi and connect 10 devices to it. Those 10 devices are "private networks," and the main one (the router-facing connection) is the "public network." When Device 1 asks "show me YouTube," that request goes to the Wi-Fi as a private-address request, which the router translates into a public-address request on the device's behalf, sends it out, gets the response from the server, and the Wi-Fi (using its NAT table) sends that response back down to exactly the right private device that originally asked.

Why Does OOP Even Exist?

💬 The Bank Account Analogy

Imagine a bank with millions of account holders. Managing all of that data securely is critical, because money is involved — data has to be private, secure, and consistent everywhere. Managing this becomes very tough once you have millions of users: if you write one-off code for every single user, or don't carry a proper organized structure, things spiral into complexity. The same applies to a College Management System — you have students, teachers, courses, attendance, results, fees — managing all of this with raw, unorganized code becomes a nightmare. If something breaks, you can't debug it. So to get proper management at scale, we organize things — and that's exactly where OOP (Object-Oriented Programming) comes in.

🤔 The "Without OOP" Problem — His Code Walkthrough

Without OOP, you'd write something like: name = "Shivansh", marks = 90, then a separate print statement combining name + marks. Everything is scattered — you don't know where each piece lives, lines might be reordered, and if a variable gets reused/overwritten elsewhere (e.g. marks accidentally set to 80 somewhere else), the entire system can fail. It's also not scalable — for 1000 students, you'd need 1000 separate variables: studentName1, studentName2... marks1, marks2... This becomes extremely complex very fast.

✅ The OOP Solution

You build one Student class containing name and marks as properties, plus a print method. Now for 1000 students, you just create instances: S1.name, S1.marks, S1.print() — clean, repeatable, and you don't manually redeclare variables each time. Data becomes fixed/structured automatically inside the object.

Real-World Objects → Programming Objects

💬 The Room Around You

Right now, sitting and teaching this lecture, there's a computer, a desk, a pen, a pen tab, a camera around him — all of these are real-world objects. The exact same concept of "object" carries over into programming. Just like his laptop has a keyboard connected, a camera connected, a light connected — and all those connections need to be organized properly, or something might randomly stop working (the light could go off, the camera might not connect properly) — organizing connections/relationships properly is exactly the discipline OOP brings to code.

Benefits of OOP

  • Code reuse — write a function/class once, use it 50 places. Increases usability, and it's not complex — change something in one place and it reflects everywhere it's used, in different ways with different arguments.
  • Security — sensitive data stays in one place → this is exactly Encapsulation (covered next).
  • Extensibility — a clear folder/file organization (Models / Controllers / Routes / Utils — the classic MVC pattern) means when a new feature is needed, you know exactly where to add it.
💬 The MVC Folder-Structure Analogy

If you have a folder structure with src/ containing separate models/, controllers/, and routes/ folders — and inside each, files per feature (student model, student controller, student route) — then any time you need to change something about "student," you know exactly where to go: the student-specific files in each folder, without touching anything else. That's the whole point of OOP: organize so that you can manage, secure, reuse, and extend it.

What is OOP — Formal Definition

📖 Definition

OOP = Object-Oriented Programming. A programming method where we consider everything in terms of objects, each having some data and some behavior. Just like real-world objects have properties and behaviors, here too: data → Attributes/Properties, behavior → Methods/Functions.

💬 The Car Analogy

A car is the object. Its properties are color and speed. Its methods are: you can brake, you can drive it. The model, price, color are its attributes/properties; the way it accelerates, brakes, signals — that's its methods/behavior.

🧠 Memory Trick — "EAPI"

Since you build APIs in backend work, remember it as E-A-P-I: Encapsulation, Abstraction, Polymorphism, Inheritance.

1. Encapsulation

📖 Definition

Wrapping data and methods into a single unit, and restricting direct access to that data.

💬 The Capsule Analogy

Think about it from the name itself — like an actual medicine capsule. You've put all the data of one type into a single "capsule," restricting its access so it's not freely available to everyone. Only whoever has the rights gets to access it; it's usable exactly where it's needed, not exposed everywhere. Just like a medicine capsule should only be consumed by the person who's actually sick — not by just anyone who comes along.

🤔 Why this matters — His Modular Architecture Example

If I need to change something about "student," that change should stay contained to the student model/routes — it shouldn't break professor, fees, course, or attendance logic elsewhere. This containment is the entire point of encapsulation: localize changes, restrict exposure.

💬 E-commerce & Banking Real-World Examples

Imagine an e-commerce site (like Amazon) with 100 registered sellers — one seller obviously shouldn't be able to see another seller's data. Each seller should only see their own customers' data and their own products' data. Similarly in banking: a user should only be able to access their own account using their own PIN — that's data hiding, security, and controlled access all working together via encapsulation.

His Bank Account Code Walkthrough

A BankAccount class has an amount attribute, a deposit() method that adds to the amount, and a getBalance() method that returns the current amount. Walking through it with real numbers: starting balance ₹100, deposit ₹10 → deposit() runs → balance becomes ₹110 → getBalance() now returns ₹110. Multiple users can each have their own account object, and one user's deposit method should never be callable to drain or view another user's balance — that's controlled access in action.

2. Abstraction

📖 Definition

Hiding implementation details and showing only the essential features.

💬 The Car-Driving Analogy

You know a car has an engine. But do you care how the engine spins, how it rotates the wheels? No — you just need to turn the steering wheel and press the accelerator, and the car moves. You don't need to know how pressing the accelerator makes the wheel spin, how the brake makes the car stop, how hydraulics/friction work internally. You just understand: press accelerator → car moves forward; press brake → car stops; turn steering → car turns. That's it — and that alone gets your work done. The car's outer panel/hood hides the engine, the hydraulic brakes, the whole steering mechanism — from outside it just looks like a clean car. That hidden complexity is fine as long as the essential behavior (engine must exist, must work) is properly implemented underneath — you just don't need to show it.

💬 The ATM Machine Analogy

You go to an ATM, insert your card, and cash comes out after entering your PIN. You don't see how many internal API calls happened, how many function calls fired, how many security checks ran on your card, how the machine physically counts and dispenses the cash. You only see: insert card + PIN → screen → keypad → card slot → cash. That's abstraction: hide all internal logic, show only the input/output the user actually needs.

🤔 Code-level Abstraction Example

You build a "hidden" file containing all the classes, functions, objects, attributes fully implemented. Then, in a separate file, you import that hidden file and just call car.drive() or card.print() directly — using the class's instance-creation methods without ever showing how the class itself was defined internally. You're not showing the user "how to build a class" — you're just giving them a few methods (in whatever readable language) and hiding the rest of the implementation.

3. Inheritance

📖 Definition

One class acquires properties of another class.

💬 The Parent/Child Resemblance Analogy

You "inherit" something from your father — maybe his eyes, maybe you resemble him in some way. You acquired that trait. Same idea here: a class can acquire/extend the properties of another class. If a Parent class has certain variables and methods, the Child class's object can behave the same way — using everything from the parent's blueprint, plus its own additions.

📝 Important asymmetry to remember

Parent classes cannot call Child class members. But Child classes can call Parent class members. (Parent → Child: no. Child → Parent: yes.)

💬 The Banking System Example

Imagine you want a Customer entity and a Bank entity. Bank has properties/methods like giving loans, generating transaction IDs, collecting payments, updating details. Animal has an eat() method — all animals eat. Dog extends Animal, and a Dog object can directly call animal.eat() even though "eat" wasn't redefined in Dog — it's accessible purely because of inheritance — and Dog also has its own unique bark() method that Animal never had.

4. Polymorphism

📖 Definition (Etymology-based)

Poly = multiple (like a polymer = a chain of repeating "multiple" units). Morph = form/structure. So Polymorphism = "many forms" — one name, many different forms/behaviors.

🤔 The "addition" Example

Say you had an add function that takes two arguments and sums them. But you actually want add to work no matter how many arguments you pass — 2 numbers, 5 numbers, 100 numbers, always sum them all. You build a common add function that loops/sums based on however many arguments came in, and returns the total. So calling it with 2 args or 100 args both "just work" — same function name, multiple usage forms. That's the spirit of polymorphism.

🔥 The Two Types of Polymorphism

🔥 Memory trick

Compile has an "L" → Loading → Method Overloading. Runtime has an "R" → Run → Overrriding. (Compile-time → Overloading. Run-time → Overriding.)

📖 Compile-Time → Method Overloading

Same method name, different parameters (different count or types of arguments). Decided at compile time. Example: two add() methods in a MathUtils class — one taking 2 arguments, one taking 3 — both named add, resolved based on how many args are passed.

📖 Run-Time → Method Overriding

A Child class redefines a method that already exists in the Parent class, with the same name. Example: Animal has sound() that prints something generic; Dog (extends Animal) redefines sound() with its own print statement — and at runtime, calling it on a Dog object resolves to Dog's version, not Animal's.

Relationship Types — Association, Aggregation, Composition

📖 Association

A general relationship between two classes where they're connected but exist independently — no ownership. Can be one-to-one, one-to-many, or many-to-many.

💬 Teacher–Student Example

A Teacher can exist without a Student, and a Student can exist without a Teacher — but there's a relationship: the teacher teaches, the student studies. They're independently existing objects, just connected.

📖 Aggregation — "Has-A" (Weak)

A weak "has-a" relationship. The child can exist independently of the parent. Represented in UML by a hollow diamond.

💬 Car–Engine Example

A Car "has-a" Engine. But the Engine can also exist independently — it's not built inside the car class permanently; you could remove this engine and use it elsewhere (in a different car, a generator, etc.). Removing the car conceptually doesn't destroy the engine.

📖 Composition — "Has-A" (Strong)

A strong "has-a" relationship with full ownership. The child cannot exist independently — its life cycle is tied directly to the parent's. Represented in UML by a filled/solid diamond.

💬 Human–Heart Example

A Human "has-a" Heart — but unlike the engine, the heart is created directly inside the Human, and if the Human is destroyed, the Heart is destroyed too. A heart can't just float around independently outside a human object. This is much stricter than the car/engine case.

Keys in DBMS

🔥 Extremely hot interview topic

"This is used a lot in database design, and interviews ask this a lot — a really hot topic."

Super Key

📖 Definition

Any attribute (or set of attributes) that can uniquely identify a tuple in a relation. Could be a single attribute or several combined — e.g. Employee ID alone is a super key, but Employee ID + Name combined is also technically a (redundant) super key.

Candidate Key

📖 Definition

A minimal super key — you can't remove any attribute from it without losing uniqueness. There can be multiple candidate keys per table (e.g. Employee ID alone, or Email alone, or Mobile Number alone — each independently unique).

🔥 Primary Key

📖 Definition

The candidate key chosen by the database designer to uniquely identify tuples. Must be unique, never NULL, and act as the table's main identifier.

Alternate Key

📖 Definition

Any candidate key that was not chosen as the primary key. (If you had 4 candidate keys and picked 1 as primary, the remaining 3 become alternate keys.)

🔥 Foreign Key

📖 Definition

An attribute (or set of attributes) in one table that references the Primary Key of another table — this is exactly how you connect/join tables. E.g. an Employee table stores departmentId, which is a foreign key referencing the Department table's primary key, departmentId.

Composite Key

📖 Definition

A primary key made of two or more attributes combined.

Surrogate Key

📖 Definition

An artificially generated key, typically an auto-incrementing integer — basically your classic "serial number" column (1, 2, 3...), which auto-increments and may skip numbers if rows get deleted.

🔥 SQL Query Execution Order

🔥 Very important — interviewers test if you actually understand this, not just write queries
Fig 16 — Real SQL execution order. SELECT runs almost last, not first!
🤔 Why does this matter? — His Exact Example

This is why you cannot use a SELECT alias inside a WHERE clause — because WHERE is evaluated before SELECT ever runs. Example: SELECT * FROM student WHERE age >= 15 — even though SELECT is written first in the query text, the engine actually processes FROM → WHERE → ... → SELECT. So WHERE has no idea about anything SELECT would have computed, because WHERE runs first. If you've written a query like: a JOIN between two tables happens first, then any GROUP BY filtering, then HAVING, and only at the very end does SELECT pull out the actual columns, followed by LIMIT/OFFSET/ORDER BY at the very last step.

JOINs

📖 Why JOINs exist

To combine column data from two or more tables — e.g. an Employee table storing only a departmentId, and a separate Department table holding the actual department name — a JOIN lets you pull the department's name alongside each employee's row.

-- INNER JOIN: only rows that match in BOTH tables
SELECT e.name, d.department_name

INNER JOIN department d ON e.department_id = d.department_id;

SELECT e.name, d.department_name

LEFT JOIN department d ON e.department_id = d.department_id;

SELECT e.name, d.department_name
FROM employee e

TCL — Transaction Control Language

🤔 Why TCL/Rollback matters — His Bank Example

Imagine a transaction that involves 4 sequential SQL operations across multiple tables, taking 50 seconds total. If the database shuts down 10 seconds in (say, only the first two operations completed), you can't leave the data half-updated — e.g. money deducted from one account but never reflected in the UPI app. You must roll back everything. So a transaction must either fully complete, or fully roll back — never half-done.

BEGIN;
UPDATE account SET balance = balance - 500 WHERE id = 1;
UPDATE account SET balance = balance + 500 WHERE id = 2;
COMMIT;   -- or ROLLBACK; if something fails

Window Functions

  SUM(sales) OVER (ORDER BY sale_date) AS running_total

Window functions like RANK(), DENSE_RANK(), LAG(), LEAD() let you do calculations across a set of rows related to the current row, without collapsing them into one group (unlike GROUP BY).

WITH high_earners AS (
  SELECT * FROM employee WHERE salary > 100000
)

Defines a temporary named result set inside a query — makes a complex query far more readable, and supports recursive calls.

UNION vs UNION ALL

SELECT city FROM customers
UNION
SELECT city FROM suppliers;   -- removes duplicates

SELECT city FROM customers
UNION ALL
SELECT city FROM suppliers;   -- keeps duplicates

EXISTS vs IN

SELECT * FROM student s
WHERE EXISTS (SELECT 1 FROM enrollment e WHERE e.student_id = s.student_id);

SELECT * FROM student
WHERE student_id IN (1, 2, 3);
🤔 The difference

EXISTS returns just a yes/no and is generally more efficient for large subqueries — it stops as soon as one match is found. IN compares against a fixed list of values directly and pulls back every matching value with SELECT.

05 · Normalization & ACID 4:58:26 – 5:23:59

Normalization

📖 Definition

The process of organizing a relational database to reduce data redundancy (repetition) and eliminate data anomalies.

🤔 Why does normalization matter? — His Unnormalized Table Example

Picture a single flat table holding orders, with a "products" column crammed full of repeated/multiple values. Problems that show up:

  • Insert Anomaly — to add a new product to a catalog, you'd be forced to create a fake order just to have somewhere to put it.
  • Update Anomaly — if Rahul's phone number changes, you must update it in every single row where Rahul appears — miss even one, and the data becomes inconsistent.
  • Delete Anomaly — delete order #3 (which happens to be the only row mentioning a particular keyboard product), and you accidentally erase all knowledge that the keyboard product ever existed.

1NF — First Normal Form

📖 Rule

Every column must contain only atomic values — no repeating groups, no arrays/lists crammed into one cell.

Example violation: a "courses" column holding "OS, CN, DBMS" as one comma-separated string for a single student. Fix: split into a separate row per course (or a separate join table), so each cell holds exactly one atomic value — if you need OS specifically, you can query it directly without parsing a packed string.

2NF — Second Normal Form

📖 Rule

Must satisfy 1NF, AND every non-key attribute must be fully functionally dependent on the entire primary key (no partial dependency).

🤔 His Enrollment Table Example

An Enrollment table with composite key (student_id, course_id). If student_name depends only on student_id (not the full composite key) and course_name depends only on course_id — that's a partial dependency, violating 2NF. Fix: split into three smaller tables — Student, Course, and Enrollment — each holding only what truly belongs together.

3NF — Third Normal Form

📖 Rule

Must satisfy 2NF, AND have no transitive dependency — a non-key attribute should not depend on another non-key attribute.

🤔 Transitive Dependency Example

If zip_code determines city, and city is stored as a column alongside zip_code in a student table where neither is the primary key — that's a transitive dependency (key → zip_code → city). Pull city out into its own lookup, tied directly to zip_code.

BCNF — Boyce-Codd Normal Form

📖 Rule

A stricter version of 3NF: for every functional dependency X → Y, X must be a super key.

4NF

📖 Rule

Must satisfy BCNF, with no multi-valued dependency.

Transactions & ACID Properties

🔥 Extremely important interview topic
📖 What is a Transaction?

A logical unit of work made up of more than one data operation — they must all succeed together, or all fail together.

💬 His Bank Transfer Example, Revisited

4 operations need 5 minutes total. If the system crashes after 3 minutes, 3 of the 4 operations completed and 2 didn't. You now must roll back the 3 completed operations — either the whole thing succeeds, or none of it does. If you don't roll back properly, you can get scenarios where money is deducted from one account but the corresponding deposit (or the database update reflecting it) never actually went through — leaving the user's account and the bank's books in an inconsistent state.

Fig 18 — The 4 ACID properties

Atomicity

📖 Definition

A transaction must be treated as a single, indivisible unit. If even one part fails, undo everything — never leave it half-done.

Consistency

📖 Definition

A transaction brings the database from one valid state to another valid state. E.g. account balance can never go below zero — and every account in the system follows this same rule consistently.

Isolation

📖 Definition

Concurrently running transactions should not interfere with each other — each runs as though it's the only transaction happening, until it's fully complete. No transaction should be able to peek into another's intermediate/uncommitted state.

Durability

📖 Definition

Once a transaction is committed, it stays committed — even if the system crashes immediately after. You won't lose committed data; you'd build replicas/handle failures so this holds.

Transaction States

Fig 19 — Transaction lifecycle: Begin → Active → Partially Committed → Committed (or → Failed/Aborted → rollback to Begin)

Concurrency Control

🤔 The Concurrency Problem

One transaction is reading data while another updates it mid-read — the reader ends up working with stale data. If a value was 10 and someone changes it to 15 mid-read, whoever read it before the change still has the outdated "10" — leading to inconsistent results unless properly isolated.

Locking Mechanisms

Lock TypeBehavior
Shared LockMultiple transactions can read simultaneously; none can write until released
Exclusive LockOnly one transaction can hold it — for writing — and no one else can read or write until it's released

Two-Phase Locking (2PL)

PhaseBehavior
Growing PhaseTransaction can acquire new locks, but cannot release any
Shrinking PhaseTransaction can release locks, but cannot acquire any new ones

Deadlock in Concurrency Control

🤔 How it happens here

T1 holds a lock on A and waits for B; T2 holds a lock on B and waits for A — both stuck in an infinite loop, neither can proceed. Resolved with timestamp-based methods (e.g. Wait-Die) — if a lock is held beyond a certain time, it's force-released and the transaction restarts.

Indexing

🤔 Why indexing exists

If a table has 1 crore (10 million) rows, you can't scan every single row to find one record — you need something faster. Indexing solves this.

💬 The Dictionary / Book-Chapter Analogy

When you open a dictionary, the top of each page shows a guide word, so you know "everything starting with C is around here" — you don't read every single word from page 1. Similarly, in a book, you flip to the index and see "Chapter 4 starts on page 50," so you jump straight there instead of reading the whole book to find it.

📖 Definition

An index builds a B-Tree structure over a column, so lookups happen in O(log n) time instead of a full O(n) table scan.

CREATE INDEX idx_student_name ON student (name);

🔥 Direct comparison question

Clustered determines the physical order of data storage in the table itself. Non-Clustered keeps a separate structure entirely from the table.

B-Tree Indexing

📖 Why B-Trees specifically

Most relational databases use B-Trees — self-balancing trees where each node can have multiple children, holding different data. Any record can be reached by traversing one tree, efficiently, in O(log n) time, because of the balanced/binary-search-like structure.

Hashing in Databases

📖 Definition

A hash function maps a key to a specific bucket/location, so you can jump directly to where the data should be instead of searching.

💬 The Phone-Book "A-to-Z, Page Range" Analogy

Like that classic phone-book example: "A" starts on page 3, "B" starts on page 5 — you know exactly where to jump. You can further divide within "A" — A-to-A starts here, A-Z ends there. With enough data, you build a range-based table for exactly this kind of hash-style lookup.

🤔 A worked example — 4 buckets, 100 records

Divide 100 records into 4 buckets of 25 each (using a hash like "divide by 4"). Each bucket then holds its own 25-record slice, and lookups jump straight to the relevant bucket instead of scanning all 100.

Dynamic Hashing extends this idea further, growing/shrinking buckets as data scales — less critical for interviews than the core hashing concept itself.

  • Kernel = always-resident core. Types: Monolithic, Micro, Hybrid, Exo.
  • Process = active, dynamic, has own memory. Program = passive, static, on disk.
  • Thread = lightweight, shares code/data, own stack/registers. One thread crash can kill the process.
  • 5 Process States: New → Ready → Running → Waiting/Terminated.
  • Scheduling: FCFS (convoy effect), SJF (starvation), SRTF (high overhead), Round Robin (quantum trade-off), Priority (aging fixes starvation), Multilevel Feedback Queue (most flexible, most complex).
  • Deadlock = MHNC: Mutual Exclusion, Hold & Wait, No Preemption, Circular Wait — all 4 needed simultaneously.
  • Deadlock handling: Prevention, Avoidance (Banker's Algorithm — safe state check), Detection & Recovery, Ignorance (Ostrich Algorithm — used by Linux/Windows).
  • Mutex (binary, 0/1, only locker unlocks) vs Semaphore (integer, wait/signal, counting).
  • Paging = fixed size, internal fragmentation, transparent. Segmentation = variable size, external fragmentation, user-visible.
  • Virtual Memory = disk used as RAM extension; page fault when page not in RAM. Replacement: FIFO, Optimal, LRU, Clock.
  • Thrashing = more time paging than executing.
  • Memory hierarchy (fastest→slowest, costliest→cheapest): Registers → Cache → RAM → SSD/HDD → Network.

◈ Computer Networks

  • Network = interconnected devices sharing data/resources.
  • Range tiers: PAN (10m) → LAN (building) → MAN (city) → WAN (world/internet).
  • Topologies: Bus (1 cable, single point of failure), Star (central hub, most common), Ring (1-direction loop), Mesh (most fault-tolerant), Tree (Bus+Star), Hybrid.
  • OSI's 7 layers (Physical→Application) — "All People Seem To Need Data Processing." TCP/IP's 4 layers compress these.
  • Routing = Layer 3 (Network). Switch = Layer 2. SSL/TLS = Layer 6 (Presentation).
  • TCP = reliable, 3-way handshake (SYN→SYN-ACK→ACK), slower. UDP = unreliable, no handshake, faster (video/gaming/DNS).
  • HTTP=port 80, HTTPS=port 443 (HTTP+SSL), SMTP=25, IMAP=143, RDP=3389.
  • DNS = domain name → IP address translator (central lookup database).
  • DHCP / DORA: Discover → Offer → Request → Acknowledge — auto-assigns IP/subnet/gateway/DNS.
  • ARP = IP → MAC. RARP = MAC → IP (legacy).
  • MAC address = 48-bit, unique per device (NIC hardware).
  • Hub (L1, broadcasts to all) vs Switch (L2, MAC-targeted) vs Router (L3, IP-based).
  • NAT = private IP ↔ public IP translation. VPN = encrypted tunnel, masks real IP/location.
  • Duplex: Simplex (1-way, TV), Half (1-at-a-time, walkie-talkie), Full (simultaneous, video call).

◆ OOP Systems

  • OOP exists to organize code via objects (data + behavior) instead of scattered procedural variables/functions — gives reuse, security, extensibility.
  • Class = blueprint/template. Object = real instance built from it (Car class → Ferrari/Sumo/Nano objects).
  • 4 Pillars = "EAPI": Encapsulation, Abstraction, Polymorphism, Inheritance.
  • Encapsulation = wrap data+methods, restrict access ("capsule" — security/data hiding).
  • Abstraction = hide implementation, show only essentials (car's engine hidden, just drive it — convenience).
  • Inheritance = one class acquires another's properties. Types: Single, Multilevel, Hierarchical, Multiple, Hybrid. Parent can't call Child; Child can call Parent.
  • Polymorphism = many forms, one name. Compile-time = Overloading (same name, diff params). Run-time = Overriding (child redefines parent's method).
  • Abstract Class (concrete+abstract methods, no multiple inheritance, has constructors) vs Interface (abstract methods only, supports multiple inheritance, no constructors).
  • Association (generic, no ownership, e.g. Teacher–Student) → Aggregation ("has-a," weak, independent lifecycle, e.g. Car–Engine, hollow diamond) → Composition ("has-a," strong, dependent lifecycle, e.g. Human–Heart, filled diamond).
  • Access Modifiers: Public (anywhere), Private (same class), Protected (package+subclass), Default (package only).

▦ DBMS

  • Data = raw, no meaning. Information = processed, contextualized data.
  • DBMS = software managing a database (CRUD + security + concurrency) — solves File System's redundancy/inconsistency/security problems.
  • 3-Schema Architecture: External (user view/GUI) → Conceptual (logic) → Internal (physical storage).
  • ER Model: Entity (rectangle), Attribute (simple/composite/derived), Relationship (unary/binary/ternary); Cardinality: 1:1, 1:N, N:1, N:M.
  • Keys hierarchy: Super Key ⊇ Candidate Key (minimal) ⊇ Primary Key (chosen, unique, never NULL); rest = Alternate Keys. Foreign Key references another table's Primary Key (enables joins). Composite Key = multi-attribute. Surrogate Key = auto-increment ID.
  • DDL (CREATE/ALTER/DROP/TRUNCATE) · DML (INSERT/UPDATE/DELETE) · DCL (GRANT/REVOKE) · TCL (COMMIT/ROLLBACK) · DQL (SELECT).
  • DROP (structure+data) vs TRUNCATE (data only, keeps structure) vs DELETE (specific rows, supports WHERE + rollback).
  • SQL execution order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT (NOT the order it's written in!).
  • JOINs: INNER (match only), LEFT (all left+match), RIGHT (all right+match), FULL (everything+NULLs), CROSS (Cartesian), SELF (table joined with itself).
  • Normalization: 1NF (atomic values) → 2NF (no partial dependency) → 3NF (no transitive dependency) → BCNF (determinant must be super key) → 4NF (no multi-valued dependency).
  • ACID: Atomicity (all-or-nothing), Consistency (valid state→valid state), Isolation (no interference), Durability (committed = permanent).
  • Indexing = B-Tree structure for O(log n) lookup instead of full scan. Clustered (physical order, 1 per table) vs Non-Clustered (separate structure, many allowed).
  • SQL (fixed schema, vertical scale, full ACID) vs NoSQL (flexible schema, horizontal scale, eventual consistency) — MySQL/Postgres vs MongoDB/Cassandra.
  • CAP Theorem: in a partition-tolerant system, pick Consistency OR Availability — not both.
